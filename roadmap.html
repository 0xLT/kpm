kpm Roadmap

An npm-style package manager for wiki-linked Markdown knowledge bases. The roadmap runs on two tracks: Substrate & Distribution (the supply side — packaging, versioning, composition) and Agent Leverage (the demand side — retrieval, tools, and an SDK that let agents actually use the knowledge). Track B builds on Track A's primitives

Dependency: Track B's NOW assumes Track A's NEXT primitives — the MCP serve surface and the local query/retrieval index. Start Track B once those land (or co-develop them). All Track B items are planned (none in flight yet).

Model boundary (invariant #1 preserved): kpm's retrieval and context primitives are deterministic and never call a model. Only the opt-in example agents and the eval harness call models, and they live outside the core. The substrate stays reproducible.

🧭 Guiding principles (invariants)

  1. The core never calls an LLM API. Synthesis stays delegated to spawned CLIs/adapters — reproducibility is the thesis. (Track B respects this: retrieval/context primitives are model-free; only opt-in example agents + the eval harness call models, outside the core.)
  2. Crisp boundary. kpm owns mechanics (resolution, locking, hydration, copy, wikilink rewrite, validation, packaging); LLM CLIs / agents own synthesis & reasoning; authors write Markdown.
  3. Server-less first. Registry/discovery/search/hosted-retrieval must work with Git, static JSON, and local indexes before any hosted service.
  4. One substrate, two interfaces. Humans (CLI + Obsidian) and agents (MCP + SDK) read the same vault and lockfile.
  5. Determinism & integrity are sacred. Content/tarball hashing, strict wikilinks, lockfile-as-truth don't regress for convenience.
  6. Strict, but with great errors. Every strictness ships with actionable remediation pointing at the command that fixes it.

🚫 Non-goals / guardrails

  • Core never embeds an API key or calls a model directly.
  • No hosted infrastructure in Now/Next — server-less first.
  • Don't regress determinism, integrity, or strictness for convenience.
  • Singleton-by-name stays the default even after coexistence ships.
  • Not a full npm replacement; not an Obsidian Sync replacement.
  • Not a hosted Markdown site generator — export adapters that rewrite link format are in scope; shipping kpm as an SSG is not.
  • No security sandboxing for untrusted content; audit stays advisory.
  • No automatic LLM synthesis that silently modifies package content.
Track B additionally
  • Retrieval/context primitives stay model-free; only opt-in agents + eval call models.
  • No hosted retrieval service in Now/Next.
  • Agent write-back never silently mutates packages — proposals only.

🔁 Cross-cutting (continuous)

  • Strong vitest coverage for every new command, including error paths.
  • Docs updated per feature; knowledge.json / lock schema changes stay backward-compatible or bump lockfileVersion deliberately.
  • Keep resolution → hydration → composition → synthesis distinct; add a structured command parser before the surface grows unwieldy.

🎯 Success metrics

Track A — Substrate
  • A new user installs from npm and completes the quickstart without cloning this repo.
  • A user installs at least one real GitHub-hosted knowledge package.
  • doctor / pack give accurate, actionable feedback; compose is deterministic with no stale output.
Track B — Agent Leverage
  • An agent answers over an installed vault with correct citations to package@version.
  • kpm eval shows a measurable task-success lift from vault context.
  • A mainstream agent framework mounts a kpm vault as a retriever in < 10 lines.
